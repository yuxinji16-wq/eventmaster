## 3. 工程化治理目标

### 3.1 三阶段目标

| 阶段 | 目标 | 验收标准 |
|------|------|----------|
| 稳定化 | 恢复绿色基线，消除明显契约错误 | 后端、前端测试和构建全部通过 |
| 标准化 | 统一接口、目录、文档、状态和验证流程 | 新功能可按固定模板落地并可回归 |
| 生产化 | 具备部署、迁移、安全、监控、备份和发布能力 | 可支持试点上线和可控回滚 |

### 3.2 核心治理原则

1. 单一主线：前端主线为 `frontend/`，后端主线为 `backend/app/`。
2. 契约先行：API 变更必须先明确请求、响应、错误码和权限。
3. 测试门禁：任何代码交付不得绕过失败测试。
4. 文档同步：文档状态必须反映实际代码和验证结果。
5. 数据单源：业务数据只来自后端 API，前端不得用本地存储或 mock 数据作为业务来源。
6. 渐进治理：优先修复阻断质量基线的问题，再处理结构性债务和生产化能力。

## 4. 目标架构约定

### 4.1 前端主线

主目录：`frontend/src`

职责分层：
- `pages/`：路由级页面。
- `components/`：业务组件。
- `shared/`：共享 UI 组件。
- `services/`：API 客户端和外部服务。
- `utils/hooks.ts`：API 数据获取和前后端字段适配。
- `context/AuthContext.tsx`：认证状态。
- `constants/`：导航、枚举、权限配置。
- `types/`：前端业务类型。

必须治理的问题：
- `backendApi.ts` 中的 `API_BASE` 应改为环境变量配置。
- `ProtectedRoute` 必须实际包裹受保护路由。
- 已删除未被引用的 `apiCompatible.ts` 兼容 mock 层，默认业务路径只允许走真实后端 API。
- 业务页面中的 `alert()` 应统一替换为 Toast。
- 仪表盘等页面不得在 API 失败时降级为演示数据。
- 列表查询参数由 `backendApi.ts` 集中适配，前端页面语义保留 `search/stage`，对后端统一发送 `keyword/status`。

### 4.2 后端主线

主目录：`backend/app`

职责分层：
- `routers/`：HTTP 路由和依赖注入。
- `schemas/`：Pydantic 请求与响应模型。
- `services/`：业务规则。
- `repositories/`：数据库查询。
- `models/`：SQLAlchemy 模型。
- `core/`：配置、安全、日志、错误处理。
- `db/`：会话、Base、初始化。

必须治理的问题：
- `backend/app/main.py` 是唯一主启动入口。
- 历史旧结构 `backend/main.py`、`backend/models.py`、`backend/database.py`、`backend/schemas.py`、`backend/routers/`、`backend/services/` 已删除。
- 当前依赖 `Base.metadata.create_all()` 初始化结构，生产化前必须引入迁移机制。
- 敏感接口必须在后端执行权限校验，前端权限控制只用于体验优化。

### 4.3 API 契约边界

字段命名：
- 后端 API 使用 `snake_case`。
- 前端业务类型使用 `camelCase`。
- 转换只能发生在 API adapter 或 hook 层。

列表响应已统一为后端数组契约：
- activities/materials/suppliers/opportunities 主列表接口返回数组。
- 前端 API 类型直接声明为数组响应。
- 需要分页元信息时，再以独立任务引入统一 envelope，不在页面层写兼容分支。

查询参数必须统一：
- 后端搜索参数统一使用 `keyword`。
- 前端页面层可使用 `search` 作为交互语义，但只能在 API adapter 层转换为 `keyword`。
- 商机页面层可使用 `stage` 作为漏斗语义，但只能在 API adapter 层转换为后端 `status`。
- 分页参数统一为 `skip/limit` 或 `page/page_size`，不能混用。

## 5. 质量门禁

### 5.1 最低验证矩阵

| 变更类型 | 必须验证 |
|----------|----------|
| 后端模型、仓储、服务、路由 | `cd backend && python -m pytest tests/ -q` |
| 前端组件、hooks、类型、路由 | `cd frontend && npm test` |
| 前端构建配置、依赖、入口、路由 | `cd frontend && npm run build` |
| API 契约变更 | 后端 API 测试 + 前端 adapter/hook 测试 |
| 样式和交互变更 | 前端测试 + 本地预览 |
| 文档状态更新 | 文档内容必须与实际命令结果一致 |

### 5.2 交付准入

一次变更满足以下条件才允许标记完成：
- 需求范围清晰。
- 文档已更新。
- 测试已新增或确认覆盖。
- 相关验证命令已执行。
- 失败项已修复或明确记录为已知风险。
- 没有新增业务 mock、硬编码演示数据或静默降级逻辑。

### 5.3 失败测试处理规则

1. 失败测试优先于新功能。
2. 先定位根因，再修改代码。
3. 禁止删除或放宽测试来掩盖真实缺陷。
4. 修复后必须运行失败用例所在测试文件；影响共享逻辑时运行全量测试。
